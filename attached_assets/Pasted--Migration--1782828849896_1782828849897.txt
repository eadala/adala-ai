ما أرسلته ليس الخطأ، بل هو محتوى ملف الـ Migration الذي تم توليده. هذا الملف بحد ذاته لا يظهر فيه خطأ نحوي واضح؛ فهو يقوم بـ:

* حذف عدد كبير من الفهارس (DROP INDEX).
* تعديل Default لقيم UUID في عشرات الجداول.
* إعادة إنشاء الفهارس (CREATE INDEX).

لذلك فإن سبب “Failed to validate database migrations” لا يمكن استخلاصه من هذا النص وحده.

لكن هناك شيء لافت جدًا

هذا الـ Migration ضخم جدًا، ويغير عشرات الجداول في دفعة واحدة. وهذا يزيد احتمال حدوث أحد السيناريوهات التالية:

* أحد الـ Indexes غير موجود أصلًا في قاعدة بيانات الإنتاج، فيفشل DROP INDEX.
* أحد الجداول الموجودة في الكود غير موجود في قاعدة بيانات الإنتاج.
* يوجد اختلاف (Schema Drift) بين قاعدة بيانات التطوير والإنتاج.
* توجد Migration سابقة لم تُطبق على الإنتاج.

⸻

أول شيء أتحقق منه

أول سطر مثلًا:

DROP INDEX "idx_uq_cases_office_case_number";

إذا كان هذا الـ Index غير موجود في الإنتاج، فسيفشل التنفيذ إذا لم يُستخدم:

DROP INDEX IF EXISTS ...

وينطبق ذلك على جميع أوامر DROP INDEX.

⸻

الشيء الثاني

أرى أوامر مثل:

ALTER TABLE "cases"
ALTER COLUMN "id"
SET DEFAULT (gen_random_uuid());

إذا لم يكن امتداد PostgreSQL الخاص بـ pgcrypto مفعلًا، فسيظهر خطأ مثل:

function gen_random_uuid() does not exist

وهذا سبب شائع جدًا عند الانتقال إلى بيئة إنتاج جديدة.

⸻

الشيء الثالث

إذا كانت إحدى هذه الجداول غير موجودة أصلًا، مثل:

ALTER TABLE "jlwm_data_quality"

فستفشل عملية التحقق مباشرة.

⸻

ما أحتاجه الآن

أحتاج سجل الخطأ الحقيقي (Logs) وليس ملف الـ Migration.

افتح في Replit:

Publishing → Logs

وابحث عن أول رسالة تحتوي على أحد هذه الكلمات:

* ERROR:
* relation does not exist
* index does not exist
* function gen_random_uuid
* duplicate key
* migration failed
* syntax error

وانسخها كاملة.

لا تعتمد على “Load failed”

هذه مجرد رسالة من واجهة Replit. الخطأ الحقيقي يكون في الـ Logs، وهو الذي سيحدد إن كانت المشكلة:

* Index مفقود.
* جدول مفقود.
* امتداد pgcrypto غير مفعل.
* تعارض بين مخطط قاعدة البيانات والإنتاج.
* أو سبب آخر.

من دون سجل الخطأ الفعلي، لا يمكن الجزم بسبب فشل التحقق من الـ Migration، لأن ملف الـ Migration نفسه يبدو من الناحية التركيبية صحيحًا.
