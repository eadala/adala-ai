يمكنك إرسال هذا التوجيه للمبرمج أو وكيل Replit. صيغته تركز على التحقق بالأدلة وعدم الاكتفاء بالافتراضات، وتمنع أي حلول مؤقتة قبل إثبات السبب الجذري.

⸻

P0 Engineering Directive — Clerk Production Root Cause Verification & Permanent Fix

بعد مراجعة جميع التقارير السابقة، لا أريد أي حلول مبنية على التخمين. أريد إثبات السبب الجذري بالأدلة ثم إصلاحه نهائياً.

المطلوب

نفّذ التحقيق التالي بالترتيب، ولا تنتقل للخطوة التالية قبل توثيق نتيجة السابقة.

المرحلة الأولى — إثبات السبب الجذري

تحقق من جميع النقاط التالية مع إرفاق الأدلة:

1. قيمة:
    * CLERK_SECRET_KEY
    * CLERK_PUBLISHABLE_KEY
    * VITE_CLERK_PUBLISHABLE_KEY
    * VITE_CLERK_PROXY_URL

مع توضيح:

* هل هي pk_live أم pk_test.
* هل جميعها تنتمي إلى نفس Clerk Instance.
* هل يوجد أي تعارض بين Secret Key و Publishable Key.

⸻

2. تحقق من إعدادات Clerk Dashboard.

أريد إثباتاً لـ:

* Allowed Origins
* Allowed Redirect URLs
* Production Domain
* Proxy Configuration
* Custom Domain (إن وجد)

وبيان هل:

adalahai.com

مضاف ومعتمد بالكامل.

⸻

3. افحص Middleware الخاص بـ Clerk.

تحقق أن:

* يتم إرسال جميع Headers المطلوبة.
* لا يوجد Header مفقود.
* لا يتم خلط مفاتيح Test مع Live.
* لا يتم إنشاء Proxy باستخدام Instance مختلفة.

⸻

4. افحص البناء الحالي الموجود في Production.

أريد معرفة:

* اسم ملف JS المحمل فعلياً.
* Commit SHA الذي بُني منه.
* تاريخ البناء.
* Publishable Key الموجودة داخل الـ bundle.
* هل يحتوي على:

VITE_CLERK_PROXY_URL

أم لا.

⸻

5. افحص عملية Deployment.

أريد إثبات أن:

* آخر Build وصل فعلاً إلى Production.
* لا يتم تشغيل Dist قديم.
* جميع ملفات JS الجديدة موجودة.
* لا توجد Assets ترجع 404.
* لا توجد نسخة قديمة ما زالت تُخدم.

⸻

6. افحص Cloudflare.

تحقق من:

* Cache-Control
* ETag
* Last-Modified
* CF-Cache-Status

وبيّن هل توجد نسخة قديمة يتم تقديمها من Cache.

⸻

7. افحص Runtime.

أريد معرفة أول Exception يمنع Clerk من الوصول إلى:

isLoaded = true

مع Stack Trace الكامل إن وجد.

⸻

المرحلة الثانية — الإصلاح

بعد إثبات السبب الجذري فقط:

* أصلح السبب الحقيقي.
* لا تضف Workaround.
* لا تضف Timeout جديد.
* لا تضف Skip أو Ignore.
* لا تُخفِ الخطأ.

بل أصلح المصدر الحقيقي للمشكلة.

⸻

المرحلة الثالثة — التحقق

بعد الإصلاح أريد إثباتاً بالصور أو السجلات أن:

* اختفى host_invalid.
* أصبح /api/__clerk/v1/environment يعيد 200.
* أصبح Clerk.isLoaded = true.
* اختفت شاشة “جاري التحميل”.
* الصفحة الرئيسية تعمل.
* تسجيل الدخول يعمل.
* لوحة التحكم تعمل.
* جميع Assets الجديدة ترجع 200.
* لا توجد ملفات قديمة.
* تم نشر آخر Commit على Production.

⸻

التقرير النهائي المطلوب

لا أريد عبارة “تم الإصلاح”.

أريد تقريراً يحتوي على:

* Root Cause.
* لماذا حدث.
* كيف تم إصلاحه.
* الأدلة قبل الإصلاح.
* الأدلة بعد الإصلاح.
* Commit SHA.
* Build ID.
* Deployment ID.
* أسماء ملفات الـ Bundles الجديدة.
* نتائج اختبار Production.
* تأكيد أن adalahai.com يعمل بالكامل بدون شاشة التحميل.

إذا تعذر إصلاح المشكلة، فلا تُجرِ أي تعديل إضافي، بل حدّد بدقة أول نقطة فشل تمنع التطبيق من الإقلاع، مع الأدلة التقنية التي تثبت ذلك.
