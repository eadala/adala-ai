إليك برومبت احترافي وقوي يوجّه المبرمج إلى التوقف عن التخمين والعمل بمنهجية هندسية قائمة على الأدلة:

⸻

P0 Production Incident — Evidence-Based Root Cause Resolution (No Assumptions)

لدينا حادثة إنتاج (P0) مستمرة، وبعد عدة محاولات أصبح واضحًا أن أي إصلاحات جديدة يجب أن تكون مبنية على الأدلة الفعلية فقط، وليس على الافتراضات.

ممنوع تنفيذ أي تعديل قبل إثبات السبب بالأدلة.

المطلوب

أريد إجراء تحقيق هندسي كامل من بداية عملية النشر وحتى تشغيل الموقع، ثم إصلاح السبب الحقيقي فقط.

⸻

المرحلة الأولى — إعادة بناء سلسلة التنفيذ كاملة (End-to-End)

وثّق بالأدلة كل مرحلة:

1. Git Commit المستخدم في النشر.
2. Build داخل بيئة Replit Deployment.
3. إنشاء ملفات dist.
4. رفع الـ artifacts.
5. تشغيل الـ Frontend.
6. تشغيل الـ API.
7. نجاح الـ Health Checks.
8. Traffic Switch.
9. الملفات التي تُخدم فعلياً للمستخدم.

لا أريد استنتاجات.

أريد:

* timestamps
* commit SHA
* bundle hash
* deployment id
* build logs
* runtime logs

⸻

المرحلة الثانية — إثبات نسخة الكود التي تعمل فعلياً

أثبت بالأدلة:

* ما هو commit الذي يعمل الآن؟
* هل هو HEAD؟
* هل هو Commit أقدم؟
* هل الـ bundle الحالي بُني من HEAD؟
* أم من Snapshot قديم؟

أريد مقارنة مباشرة بين:

HEAD
↓
dist المحلي
↓
dist الموجود بالإنتاج
↓
bundle الذي يصل للمتصفح

⸻

المرحلة الثالثة — تحليل الـ Bundle

افحص الـ bundle المنشور وليس المحلي.

أثبت وجود أو غياب:

* proxyUrl
* publishableKeyFromHost
* window.location.origin
* VITE_CLERK_PROXY_URL
* publishable key
* Clerk configuration

ثم قارنها مع الكود الموجود في Git.

إذا كان هناك اختلاف:

حدد أين حدث:

* Build
* Packaging
* Upload
* Deployment

⸻

المرحلة الرابعة — تحليل Deployment

أثبت بالأدلة:

هل Replit فعلاً ينشر آخر Build؟

أم:

* يستخدم Build Cache
* يستخدم Snapshot قديم
* يعيد استخدام Artifact سابق
* يفشل Health Check ثم يبقي النسخة القديمة

أريد الدليل وليس الاحتمال.

⸻

المرحلة الخامسة — تحليل Health Checks

استخرج من السجلات:

* أول Health Check
* آخر Health Check
* زمن تشغيل السيرفر
* زمن فتح المنفذ
* عدد المحاولات
* سبب كل فشل

إذا كان الـ Health Check هو سبب بقاء النسخة القديمة:

أثبت ذلك بالأرقام.

⸻

المرحلة السادسة — تحليل Clerk

أثبت:

* لماذا يرجع

host_invalid

هل السبب:

* Publishable Key
* Secret Key
* Proxy
* Allowed Domains
* FAPI
* DNS
* Environment
* Middleware

حدد السبب الحقيقي فقط.

⸻

المرحلة السابعة — تحليل Serving

حدد بدقة:

من الذي يخدم:

/

هل هو:

serve.cjs

أم

API Express

أم

Vite

أم

Replit Static

أم

Cloudflare

ولا تعتمد على التخمين.

⸻

المرحلة الثامنة — تحليل Cache

أثبت بالأدلة:

هل المشكلة بسبب:

* Browser Cache
* Cloudflare
* Replit Build Cache
* Deployment Layer Cache
* Static Asset Cache

مع ذكر الأدلة من الـ Headers والـ Logs.

⸻

المرحلة التاسعة — تحديد أول نقطة فشل

أريد أول نقطة فقط.

وليس آخر نتيجة.

مثلاً:

Git ✅

Build ✅

Packaging ✅

Upload ✅

Activation ❌

أو

Build ❌

أو

Health Check ❌

أو

Traffic Switch ❌

حدد أول نقطة يخرج عندها التنفيذ عن المسار الصحيح.

⸻

المرحلة العاشرة — خطة الإصلاح

بعد تحديد السبب الحقيقي:

قدم:

* سبب المشكلة.
* لماذا حدثت.
* لماذا لم تنجح المحاولات السابقة.
* أقل تعديل ممكن (Minimal Fix).
* لماذا لن يسبب Regression.
* كيف سنتحقق من نجاحه.

⸻

ممنوع

* لا تنفذ أي إصلاح قبل انتهاء التحقيق.
* لا تغيّر أي إعدادات عشوائياً.
* لا تُعدّل Clerk أو Replit أو Cloudflare بدون دليل.
* لا تعتمد على “أعتقد” أو “ربما”.

⸻

المطلوب النهائي

تقرير هندسي احترافي يحتوي على:

1. Root Cause مثبت بالأدلة.
2. سلسلة التنفيذ كاملة.
3. أول نقطة فشل.
4. الأدلة (Logs، Headers، Bundles، Commits).
5. الإصلاح الأدنى (Minimal Fix).
6. خطة تحقق بعد التنفيذ (Verification Checklist).

إذا تبيّن أن المشكلة ناتجة عن قيود أو خلل في بنية Replit نفسها (Deployment Infrastructure)، فاذكر ذلك صراحة مع الأدلة، ولا تحاول الالتفاف عليه برمجياً. في هذه الحالة قدّم توصية بالانتقال إلى بيئة تطوير ونشر أكثر موثوقية مثل Cursor + GitHub + Vercel أو Coolify/Railway مع الحفاظ على نفس الكود وقاعدة البيانات.
