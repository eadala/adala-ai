الحقائق المثبتة

* التطبيق يعمل محلياً.
* الـ API يعمل.
* المشكلة في Frontend Production Deployment.
* أكثر من نشر لم يغيّر الـ bundle الفعلي في الإنتاج.
* ظهرت أعراض تدل على إعادة استخدام Build قديم في بيئة Replit.

المطلوب

نفّذ الإصلاح كاملاً، وليس جزءاً منه.

أولاً

افحص سلسلة النشر كاملة خطوة بخطوة:

Git Commit
↓
Build
↓
Artifact Generation
↓
Artifact Upload
↓
Artifact Activation
↓
Traffic Switch
↓
Frontend Runtime
↓
Clerk Initialization

ولا تنتقل للمرحلة التالية حتى تثبت نجاح السابقة.

⸻

ثانياً

راجع بالكامل:

* artifact.toml
* build pipeline
* serve.cjs / serve.mjs
* Static assets
* dist
* dist2
* dist3
* dist4
* أي Directory يستخدم أثناء النشر

وحدد فعلياً أي Directory يتم نشره للإنتاج.

⸻

ثالثاً

تحقق أن الـ Production يستخدم فعلاً آخر Build.

أريد إثباتاً وليس استنتاجاً.

يشمل:

* اسم bundle
* Hash
* Last-Modified
* ETag
* Build timestamp
* Commit SHA

ثم قارنها مع النسخة المحلية.

⸻

رابعاً

يُمنع الاعتماد على:

VITE_CLERK_PROXY_URL

إذا كانت قيمته تُحقن أثناء Build.

إذا كانت بيئة Replit لا تمرره للبناء فلا تعتمد عليه إطلاقاً.

يجب أن يتم اشتقاق Proxy Runtime مثل:

window.location.origin

أو أي طريقة Runtime Canonical.

⸻

المطلوب في نهاية التنفيذ

أريد تقريراً نهائياً يحتوي على:

1. السبب الجذري الحقيقي.
2. لماذا حدث.
3. لماذا لم تنجح المحاولات السابقة.
4. الملفات التي تم تعديلها.
5. سبب كل تعديل.
6. المخاطر إن وجدت.
7. نتائج جميع الاختبارات.
8. إثبات أن الإصلاح يعمل في Production وليس محلياً فقط.
9. الأدلة (Logs، Headers، Bundle Hash، HTTP Status، Commit SHA).

لا تعتبر المهمة منتهية إلا إذا أصبحت الصفحة الرئيسية، وتسجيل الدخول، ولوحة التحكم تعمل بالكامل على Production دون شاشة تحميل دائمة، مع إثبات ذلك بالأدلة.
