## 1. آلية النسخ الاحتياطي الحالية

### 1.1 النسخ على مستوى التطبيق (Application-Level)

**الملف:** `artifacts/api-server/src/modules/platform/backup.ts`

النظام ينشئ نسخة JSON تحتوي على:
- القضايا (cases)
- العملاء (clients)
- الفواتير (client_invoices)
- العقود (contracts)
- وثائق (metadata فقط — بدون محتوى الملفات الثنائية)
- المستخدمين (email, name, role فقط)
- الموارد البشرية (employees, payroll, performance, incentives)
- المحاسبة (revenues, expenses, bank_accounts, cash_advances)

**التخزين:** يُحفظ في عمود `file_data TEXT` في جدول `backup_jobs` (قاعدة البيانات نفسها)

**أنواع النسخ:**
| النوع | المسار | المستخدم |
|-------|--------|----------|
| يدوي → محفوظ في DB | `POST /api/backup/create` | جميع الخطط المدفوعة |
| تحميل فوري للجهاز | `GET /api/backup/local-download` | مجاني |
| تصدير CSV/JSON مجزّأ | `GET /api/export/{entity}` | جميع الخطط |

### 1.2 النسخ على مستوى البنية التحتية (Infrastructure-Level)

**الملف:** `infra/backup/backup.sh`

```bash
# يُشغَّل عبر: 0 2 * * * /opt/adala/infra/backup/backup.sh
pg_dump -U adala adala --no-owner | gzip -9 > "$BACKUP_FILE"
```

| الجانب | التفاصيل |
|--------|----------|
| الأداة | pg_dump + gzip -9 |
| التوقيت الموصى به | 02:00 صباحاً يومياً |
| مدة الاحتفاظ | 7 أيام |
| التشفير | ضغط فقط (gzip) — لا تشفير |
| موقع الحفظ | `/opt/adala/backups/` (محلي) |
| التحقق من السلامة | `gzip -t` بعد كل نسخة |

**⚠️ تحذير:** هذا السكريبت مكتوب لبيئة Docker خارجية. في بيئة Replit الحالية، **لا يُشغَّل هذا السكريبت تلقائياً** — لا يوجد cron فعلي مُفعَّل.

## 2. عزل بيانات المستأجرين في النسخ الاحتياطي

### ✅ ما يعمل بشكل صحيح

| الفحص | النتيجة |
|-------|---------|
| `backup_jobs` يحتوي على `office_id` | ✅ |
| GET /backup/jobs مُقيَّد بـ `office_id` | ✅ |
| GET /backup/jobs/:id/download يتحقق من `office_id` | ✅ |
| DELETE /backup/jobs/:id يتحقق من `office_id` | ✅ |
| جميع مسارات التصدير تستخدم `requireAuthWithTenant` | ✅ |
| بيانات كل نسخة مُقيَّدة بـ `tenantId` في الاستعلامات | ✅ |

### ❌ ثغرات اكتُشفت وأُصلحت

#### [CVE-BACKUP-01] — CRITICAL: backup_settings غير معزول بين المستأجرين

```typescript
// الكود القديم (خطأ):
router.get("/backup/settings", requireAuthWithTenant, async (_req, res) => {
  const rows = await db.select().from(backupSettingsTable).limit(1); // بدون office_id!
});

router.put("/backup/settings", requireAuthWithTenant, async (req, res) => {
  const rows = await db.select().from(backupSettingsTable).limit(1); // أي مستأجر يكتب فوق الآخر!
});
```

**الخطورة:** أي مستأجر يمكنه قراءة وتعديل إعدادات النسخ لمستأجر آخر.  
**الإصلاح:** تم إضافة فلترة `office_id` على GET و PUT (انظر الكود المُصحَّح أدناه).

#### [CVE-BACKUP-02] — CRITICAL: /api/import لا يُحدد office_id

```typescript
// الكود القديم (خطأ):
await db.insert(clientsTable).values({
  fullName: c.fullName,
  // ❌ office_id غائب! السجل يُدرج بدون مستأجر
});
```

**الخطورة:** البيانات المستوردة تُدرَج دون `office_id` — لا تُرى لأي مستأجر، أو تُسرب للجميع حسب منطق الاستعلام.  
**الإصلاح:** تم تمرير `tenantId` في كل عملية إدراج.

### ⚠️ ملاحظة هيكلية: pg_dump لا يدعم استعادة مستأجر واحد

نسخة `pg_dump` تحتوي على بيانات **جميع المكاتب** في ملف واحد. لا يمكن استخدامها لاستعادة مكتب واحد دون استعادة كاملة للمنصة.

**التخفيف:** النسخ على مستوى التطبيق (JSON) معزولة بالكامل لكل مكتب.

## 3. النسخ الاحتياطي لملفات التخزين

| جانب | الحالة |
|------|--------|
| ملفات الوثائق والعقود والأدلة | ⚠️ metadata فقط في النسخة، المحتوى الثنائي غائب |
| ملفات Object Storage (Replit Buckets) | ❌ غير مشمولة في النسخة |
| هيكل المجلدات لكل مستأجر | ✅ موجود في Object Storage |
| نسخة احتياطية لملفات AI المُولَّدة | ❌ غير مشمولة |

**المخاطرة:** إذا فُقد Object Storage، تُفقد جميع الملفات المرفوعة بشكل نهائي حتى مع وجود نسخة DB كاملة.

## 4. قدرة الاستعادة

### 4.1 أدوات الاستعادة الموجودة

| الأداة | الوصف | يعمل؟ |
|--------|--------|--------|
| `infra/backup/restore.sh` | استعادة DB كاملة من pg_dump | ✅ (بيئة Docker) |
| `GET /backup/dr-test` | 5 فحوصات آلية للجاهزية | ✅ |
| تحميل JSON للجهاز | استعادة يدوية عبر `/api/import` | ⚠️ جزئي |

### 4.2 فحوصات DR-Test الحالية

| الفحص | ما يقيسه |
|-------|---------|
| `last_backup` | هل توجد نسخة مكتملة أحدث من 48 ساعة؟ |
| `data_integrity` | هل بيانات JSON قابلة للتحليل؟ |
| `db_live` | هل قاعدة البيانات تستجيب؟ |
| `office_registry` | هل سجل المكتب موجود؟ |
| RPO | يُقدَّر بناء على وقت آخر نسخة |

### 4.3 تقدير RTO/RPO

| المقياس | القيمة الحالية | المستهدف |
|---------|---------------|---------|
| RPO (نقطة الاسترداد) | 24-48 ساعة (يدوي) | < 4 ساعات |
| RTO الكامل (استعادة المنصة) | 2-4 ساعات (يدوي) | < 1 ساعة |
| RTO مستأجر واحد (من JSON) | 30-60 دقيقة | < 15 دقيقة |

## 7. حماية الأسرار في النسخ الاحتياطية

| السر | موجود في النسخة؟ |
|------|----------------|
| PostgreSQL credentials | ❌ لا (pg_dump لا يتضمن كلمات المرور) |
| Clerk API keys | ❌ لا |
| OpenAI/Gemini API keys | ❌ لا |
| SMTP credentials | ❌ لا |
| Stripe keys | ❌ لا |
| Cloud backup credentials (S3) | ⚠️ محفوظة في `backup_settings.cloud_config` JSONB — مشفرة بتشفير DB لكن بدون تشفير تطبيقي |

✅ **إيجابي:** لا تحتوي أرشيفات النسخ على أسرار النظام.  
⚠️ **تحذير:** بيانات اعتماد S3/R2 الخاصة بالمستأجرين محفوظة في قاعدة البيانات كـ plaintext JSON.
